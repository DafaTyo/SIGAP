Jelaskan versi terbaru dari solusi Anda, termasuk problem utama, pendekatan solusi, dan dampak
utama yang ditargetkan.
SIGAP MBG versi terbaru adalah platform monitoring vendor SPPG berbasis Secure
Local Language LLM-as-a-Service yang berjalan di atas arsitektur hybrid/edge
computing lokal. Sistem ini dirancang untuk menggantikan pendekatan lama yang
bergantung pada input manual, cloud publik dengan latensi tinggi, dan AI generik tanpa
konteks domain.
Problem utama yang diselesaikan: Sembilan kegagalan sistemik dalam tata kelola
vendor MBG — manipulasi data distribusi, nihilnya QC otomatis, hambatan pelaporan
vertikal yang lambat, AI lama kaku dan rentan prompt injection, infrastruktur tidak
scalable, dan absennya transparansi publik berbasis data real-time.
Pendekatan solusi:
Local LLM (model ter-quantize, berjalan on-premise di edge node) memangkas latensi
inferensi dari >2 detik ke <200ms
AI Guardrails multi-layer sebagai perisai terhadap prompt injection dan manipulasi
sistem
Modul QC otomatis berbasis anomaly detection (Isolation Forest + Z-Score) untuk
validasi silang laporan distribusi harian
Dashboard publik real-time yang dapat diakses siapa pun untuk memverifikasi vendor
aktif
Target dampak terukur:
Latensi AI: <200ms per query
Akurasi deteksi fraud/anomali distribusi: >95%
Blokir prompt injection: 100%
Vendor aktif termonitor: >70% dari total SPPG nasional
Pengaduan masyarakat tertindaklanjuti: >80% dalam SLA 24 jam

Tiga hal yang benar-benar baru dibanding sistem yang sudah ada:

Tidak ada sistem pengawasan pangan pemerintah di Indonesia yang saat ini

===== PAGE 14 / 31 =====
Local quantization untuk latensi rendah:
Model LLM (berbasis arsitektur seperti Mistral atau LLaMA yang di-fine-tune)
dikuantisasi ke format GGUF/GPTQ (4-bit atau 8-bit precision) sehingga dapat berjalan
pada hardware edge node standar tanpa GPU enterprise. 
Teknik ini memangkas kebutuhan VRAM hingga 75% dibanding model full-precision,
memungkinkan inferensi <200ms pada hardware yang tersedia di lingkungan GovCloud
Indonesia.
RAG dengan corpus dialek dan domain spesifik:
Knowledge base RAG dibangun dari dua lapisan: dokumen regulasi formal (kebijakan
BGN, Perpres, Permenkes) dan corpus bahasa operasional lapangan MBG yang
mencakup variasi dialek regional. 
Retrieval dilakukan menggunakan dense vector embeddings (model embedding ter-
fine-tune untuk Bahasa Indonesia) sehingga query dalam dialek lokal tetap
menghasilkan retrieval yang akurat dari dokumen formal.
LLM Guardrails — arsitektur berlapis:
- Layer 1: Input sanitization — deteksi pola injeksi (jailbreak pattern matching,
instruction override attempts) sebelum query mencapai model
- Layer 2: Context boundary enforcement — sistem prompt di-lock; instruksi tambahan
dari user input tidak dapat menimpa system-level context
- Layer 3: Output filtering — respons model divalidasi terhadap whitelist topik dan
format yang diizinkan sebelum dikembalikan ke pengguna
- Layer 4: Audit log — setiap interaksi dengan LLM dicatat untuk forensik jika terjadi
insiden
Anomaly detection berbasis Isolation Forest + Z-Score Analysis:
Kombinasi dua metode: Isolation Forest untuk deteksi anomali multidimensional (pola
distribusi yang menyimpang secara keseluruhan) dan Z-Score Analysis untuk deteksi
penyimpangan pada variabel individual (porsi per titik, frekuensi distribusi, jarak geo-
lokasi). Keduanya berjalan secara paralel dan menghasilkan composite anomaly score
yang lebih robust dari penggunaan satu metode saja.

Security & Compliance *
Bagaimana solusi Anda menangani keamanan data dan kepatuhan?
Mitigasi prompt injection:
Empat lapisan guardrail (dijabarkan di Q10) memastikan tidak ada input dari pengguna
— baik vendor, pengawas, maupun publik — yang dapat memanipulasi perilaku model.
Ini relevan secara khusus karena sistem dapat diakses oleh puluhan ribu pengguna
dengan motivasi yang tidak seragam.
Enkripsi data:
Data-in-transit: TLS 1.3 pada semua komunikasi antara client, API gateway, dan
database
Data-at-rest: AES-256 pada seluruh dokumen yang disimpan di MinIO (NIK, NIB, foto
distribusi, laporan pengaduan)
Kepatuhan regulasi:
- UU PDP (Perlindungan Data Pribadi): data sensitif vendor hanya diakses oleh aktor
dengan kewenangan yang terdefinisi dalam RBAC; tidak ada data pribadi vendor yang
dipublikasikan tanpa consent
- Perpres SPBE: platform di-deploy di GovCloud Indonesia; data tidak pernah keluar dari
wilayah RI
Audit log immutable: seluruh aktivitas sistem tercatat dan dapat digunakan untuk
keperluan investigasi forensik
Zero-Trust Architecture:
Tidak ada aktor — termasuk admin internal — yang diberi akses default ke seluruh data.
Setiap permintaan akses diverifikasi berdasarkan identitas, peran, dan konteks request.
Prinsip least-privilege diterapkan di seluruh lapisan sistem.

Horizontal scaling — teknis:
Arsitektur modular berbasis Docker + Kubernetes memungkinkan setiap modul
(perizinan, monitoring, pengaduan, LLM service) di-scale secara independen sesuai
beban. Jika volume laporan pengaduan melonjak, hanya container pengaduan yang

Redis sebagai message queue memastikan lonjakan input (misalnya, laporan distribusi

database utama.
Ekspansi domain:
Arsitektur tiga pilar SIGAP (verifikasi, monitoring, akuntabilitas) bersifat domain-
agnostik. Modul yang sama dapat diterapkan untuk program pemerintah lain dengan
struktur vendor-beneficiary serupa: program bantuan sosial, distribusi pupuk
bersubsidi, atau pengadaan obat-obatan Puskesmas. Ekspansi ini tidak memerlukan
pembangunan ulang sistem dari nol — hanya fine-tuning LLM dan RAG knowledge base
untuk domain baru.
Ekspansi geografis:
Target awal: 34 provinsi. Infrastruktur GovCloud Indonesia sudah mendukung multi-
region deployment. Node edge dapat ditambahkan di wilayah dengan konektivitas
terbatas untuk memastikan latensi tetap rendah meski skalanya bertambah.
